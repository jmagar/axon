# Llama.cpp Gemma 4 ask smoke

## User Request

The session began with a request to create a Docker Compose setup for llama.cpp, bring it online, and download "gemma e4b". The user corrected the target to `https://huggingface.co/google/gemma-4-E4B-it`, then asked whether the GPU memory use was normal and whether Axon's `ask` path could be tested with Gemma 4 using limited context.

## Session Overview

Created a standalone llama.cpp Compose file, replaced the initially wrong Gemma 3n E4B target with Gemma 4 E4B IT via the ggml-org GGUF conversion, brought the server online, verified model health and inference, inspected GPU usage, reviewed Axon's `ask` path, and added a script-only Gemma 4 `ask` smoke harness.

5. Verified the corrected server health, `/v1/models`, and a chat completion returning `gemma 4 online`.

8. Ran syntax, ShellCheck, and live smoke verification for the script.

## Key Findings

- The requested Hugging Face repo `google/gemma-4-E4B-it` contains Safetensors files, not GGUF files. llama.cpp consumed `ggml-org/gemma-4-E4B-it-GGUF:Q4_K_M`, which is the GGUF conversion of the Google model.
- `nvidia-smi` showed RTX 4070 VRAM use was accounted for: `/app/llama-server` used 4626 MiB, `text-embeddings-router` used 1356 MiB, and desktop display processes used 62 MiB total.
- Axon's `ask` path builds retrieval context first, then validates and invokes the headless Gemini subprocess contract in `src/vector/ops/commands/ask.rs:63`.
- The prompt handed to the LLM is assembled as `Question: ... Context: ...` in `src/vector/ops/commands/streaming.rs:179`.
- Context limiting knobs already exist through environment variables such as `AXON_ASK_MAX_CONTEXT_CHARS`, `AXON_ASK_CHUNK_LIMIT`, and `AXON_ASK_FULL_DOCS` in `src/core/config/types/config.rs:295`.

## Technical Decisions

- Used `docker-compose.llama.yaml` as a standalone Compose file to avoid disturbing the existing Axon stack.
- Used `ggml-org/gemma-4-E4B-it-GGUF:Q4_K_M` rather than the Google Safetensors repo directly because llama.cpp server needs GGUF-compatible model loading.
- Kept the Gemma 4 ask integration as a script-only shim instead of changing Rust backend abstractions during this session.
- Limited the Gemma 4 test path to `AXON_ASK_MAX_CONTEXT_CHARS=20000`, `AXON_ASK_CHUNK_LIMIT=6`, and `AXON_ASK_FULL_DOCS=1` because the running llama.cpp server was configured with 12k context.

## Risks and Rollback

The script is a shim over the Gemini subprocess contract rather than a first-class backend. It is useful for smoke testing but should not be treated as production backend support. Rollback is to stop the service with `docker compose -f docker-compose.llama.yaml down` and remove `docker-compose.llama.yaml` plus `scripts/test-ask-gemma4.sh`.

## Decisions Not Taken

- Did not implement a Rust OpenAI-compatible LLM backend in this session; a script-only smoke harness was lower risk and sufficient to prove the ask path with Gemma 4.
- Did not change the running llama.cpp context/parallel settings to peak-context mode; recommendations were provided, but the user shifted to testing Axon `ask`.
- Did not stop `axon-tei` to reclaim approximately 1.3 GiB VRAM because Axon embedding/query workflows may depend on it.
