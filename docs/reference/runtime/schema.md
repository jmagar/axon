# Runtime Schema

Last Modified: 2026-07-19

The SQLite schema is owned by the store crates — each crate owns its tables
and the forward-only migrations that create them. There is no central schema
file; the source of truth is the set of migration directories under each
owning crate.

> Contract source:
> [`docs/pipeline-unification/runtime/schema-contract.md`](../../pipeline-unification/runtime/schema-contract.md).
> Authoritative generated snapshot: [`database-schema.json`](database-schema.json).
> Drift-checked by `cargo xtask schemas database`.

## Table ownership by crate

| Owner crate | Tables |
|---|---|
| `axon-jobs` | `jobs`, `job_attempts`, `job_stages`, `job_events`, `job_heartbeats`, `job_artifacts`, `provider_reservations`, `config_snapshots` |
| `axon-ledger` | `sources`, `source_generations`, `source_manifests`, `source_items`, `document_status`, `cleanup_debt`, `leases` |
| `axon-jobs` (watches) | `axon_source_watches`, `axon_source_watch_runs` |
| `axon-graph` | `graph_nodes`, `graph_aliases`, `graph_edges`, `graph_evidence`, `graph_conflicts` |
| `axon-memory` | `axon_memory_nodes`, `axon_memory_edges`, `memory_records`, `memory_links`, `memory_reinforcement`, `memory_reviews` |
| `axon-observe` | `axon_observe_events`, `axon_observe_heartbeats`, `axon_observe_provider_health` |

### Clarifications

- Memory decay state lives in `memory_records.decay_json` — no separate
  `memory_decay` table.
- Watch progress is `job_events` for `job_kind=watch` — no separate
  `watch_events` table.
- Config snapshots are `config_snapshots` (owned by `axon-jobs`) — no
  `job_config_snapshots`.

## Migration rules

- Every schema change is a numbered, forward-only migration.
- Migrations are idempotent within SQLite transaction semantics.
- Never silently drop data outside explicit reset paths.
- Empty-store bootstrap and upgraded-store migration both use the same runner.
- Schema version is recorded in SQLite.
- A failed migration leaves the previous schema usable, or marks the DB
  unusable with a clear error.
- Migrations are owned by the owning store crate (`crates/<crate>/src/migrations/`).

## Empty-cutover rule

Old local data may be wiped; old code-search/code-index tables, vector
payloads, and job rows do not need migration; `axon reset` may initialize clean
stores. This is a clean-break, empty-DB target.

## Foreign keys

SQLite FKs are enabled on every connection. Source items reference
source/generation; document status references source-item identity; graph
evidence references source ids/items; cleanup debt references source/job ids;
job events reference job id. Deletes are explicit, via reset/prune policy only.

## Known artifact drift

> The generated snapshot lives at `database-schema.json` (hyphen) and is the
> real artifact (~3500 lines, regenerated by `cargo xtask schemas database`).
> A placeholder `database.schema.json` (dot) also exists from the contract's
> original path naming — the dot-named file is a stub. The contract should be
> reconciled to the hyphen path. This is tracked as a gap in
> [`docs/pipeline-unification/delivery/documentation-contract-gaps.md`](../../pipeline-unification/delivery/documentation-contract-gaps.md).

## Source of truth

SQLite migrations are the source of truth. The generated
[`database-schema.json`](database-schema.json) is a parsed snapshot of those
migrations (table columns, indexes, foreign keys, migration provenance) —
regenerate after any migration change via `cargo xtask schemas database`.

If a migration is added, update this file, the owning crate's migration
directory, and regenerate `database-schema.json` in the same PR.
