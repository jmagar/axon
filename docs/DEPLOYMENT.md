## Scope

This guide defines how to deploy and roll back Axon safely in self-hosted environments using Docker Compose.

## Deployment Targets

- Primary: compose stack defined in `docker-compose.yaml`
- Optional runtime clients: CLI (`./scripts/axon`) and Next.js app (`apps/web`)

## Prerequisites

Required:

- Docker Engine + Compose plugin
- Valid `~/.axon/.env` file (copied from `.env.example`, or generated by `scripts/dev-setup.sh`)
- Reachable TEI (Text Embeddings Inference) service
- Reachable OpenAI-compatible LLM endpoint

Recommended:

- Sufficient disk under `AXON_DATA_DIR`
- CPU/memory headroom for worker container limits

## Day-0 Bootstrap (dev-setup.sh)

`scripts/dev-setup.sh` is the recommended first-run bootstrap for all new environments. Run it once before any other steps.

```bash
./scripts/dev-setup.sh
```

It is idempotent — safe to re-run. What it does:

1. Detects OS (Linux/macOS) and installs missing system prerequisites (`curl`, `git`, `python3`, `jq`).
2. Checks for Docker and the Compose v2 plugin.
3. Installs `rustup` and the pinned Rust toolchain (from `rust-toolchain.toml`) if missing.
4. Installs `just` (task runner) using a prebuilt binary for x86\_64/aarch64 Linux, or Homebrew on macOS.
5. Installs `lefthook` (git hooks) and optional dev tools (`cargo-nextest`, `cargo-watch`, `sccache`).
6. Checks Node.js ≥ v24 and pnpm ≥ v10; installs if missing.
7. Runs `pnpm install --frozen-lockfile` for `apps/web`.
8. Creates `~/.axon/.env` from `.env.example` if it does not exist, then backfills missing entries on reruns.
9. **Prompts for `AXON_HOME`** (default `~/.axon`, flat layout — no nested `axon/` subdir), derives `AXON_DATA_DIR="$AXON_HOME"`, and pre-creates all container volume directories under it.
10. Starts production infrastructure (`docker compose --env-file ~/.axon/.env -f docker-compose.yaml up -d axon-qdrant axon-tei axon-chrome`).
11. Installs git hooks via `scripts/install-git-hooks.sh`.

**Optional flags:**

```bash
./scripts/dev-setup.sh --build       # also compiles the release binary
./scripts/dev-setup.sh --no-docker   # skip Docker checks and container startup
```

After the script completes, edit `~/.axon/.env` to fill in any `CHANGE_ME` values (TEI\_URL, OPENAI\_\*, TAVILY\_API\_KEY).

## Configuration

Populate `~/.axon/.env` before first deploy. `dev-setup.sh` handles secrets and service URLs automatically; you must supply the external service URLs manually.

### Required for all features

- `AXON_DATA_DIR` — root for all persistent volume data
- `QDRANT_URL` — Qdrant vector store URL
- `TEI_URL` — text embedding service URL (runs as `axon-tei` in `docker-compose.yaml`)
- `OPENAI_BASE_URL`, `OPENAI_API_KEY`, `OPENAI_MODEL` — LLM endpoint

### Web UI

- `AXON_WEB_API_TOKEN` and `NEXT_PUBLIC_AXON_API_TOKEN` — must be equal; auto-generated by `dev-setup.sh`
- `AXON_WEB_ALLOWED_ORIGINS` — optional allowed-origins header

### Gemini headless LLM

| Variable | Default | Purpose |
|----------|---------|---------|
| `AXON_HEADLESS_GEMINI_CMD` | `gemini` | Gemini CLI command used for synthesis. Path-like values are validated before launch. |
| `AXON_HEADLESS_GEMINI_HOME` | process `HOME` | Source HOME for copying Gemini auth into an isolated temporary HOME. |
| `AXON_LLM_COMPLETION_CONCURRENCY` | `4` | Max concurrent Gemini headless completions. |
| `AXON_LLM_COMPLETION_TIMEOUT_SECS` | `300` | Per-completion timeout. |

### Test infrastructure

This is backfilled automatically by `dev-setup.sh` and points to the current lite test infrastructure:

| Variable | Default |
|----------|---------|
| `AXON_TEST_QDRANT_URL` | `http://127.0.0.1:53335` |

### CLI binary override

`AXON_BIN` overrides which binary `scripts/axon` invokes. Useful in CI or when the binary is built from a workspace mount:

```bash
AXON_BIN=/workspace/axon_rust/target/release/axon
```

### Optional tuning

- Queue name overrides (`AXON_*_QUEUE`)
- Worker tuning (`AXON_INGEST_LANES`, `AXON_EMBED_DOC_TIMEOUT_SECS`, etc.)
- Watchdog and performance settings

Ensure `.env` is never committed. `.env.example` remains tracked.

## Standard Deploy Procedure

1. Run bootstrap (first time only, or after pulling a new checkout):

```bash
./scripts/dev-setup.sh
```

2. Pull/build images:

```bash
docker compose --env-file ~/.axon/.env -f docker-compose.yaml build
```

3. Start the stack:

**Local service deployment** (infra in Docker, app stack supervised locally):

```bash
docker compose --env-file ~/.axon/.env -f docker-compose.yaml up -d axon-qdrant axon-tei axon-chrome
```

To run the `axon` service through Compose, start the full stack:

```bash
docker compose --env-file ~/.axon/.env -f docker-compose.yaml up -d
```

The Compose `axon` service publishes MCP HTTP on `127.0.0.1:8001` by default.
Set `AXON_MCP_HTTP_PUBLISH=0.0.0.0:8001` in `~/.axon/.env` only when
intentionally exposing it beyond the host and `AXON_MCP_HTTP_TOKEN` is set.

**Local dev mode** (infra in Docker, app stack supervised locally):

```bash
docker compose --env-file ~/.axon/.env -f docker-compose.yaml up -d axon-qdrant axon-tei axon-chrome

# Start the full local stack supervisor:
cargo run --bin axon -- serve

# Or via just:
just dev
```

`axon serve` now verifies required infra health, then supervises the bridge backend, MCP HTTP server, local workers, shell server, and Next.js dev server. `just dev` is a thin wrapper that starts infra first and then runs `axon serve`.

4. Verify health:

```bash
docker compose --env-file ~/.axon/.env -f docker-compose.yaml ps
./scripts/axon doctor
```

5. Smoke test:

```bash
./scripts/axon scrape https://example.com --wait true
./scripts/axon status
```

6. Observe workers:

Workers run in-process locally — output is in the terminal directly. For infra logs:

```bash
docker compose --env-file ~/.axon/.env -f docker-compose.yaml logs --tail=200 axon-qdrant axon-tei axon-chrome
```

## Validation Checklist

- All infra containers report healthy (qdrant, tei, chrome).
- `doctor` passes critical services.
- At least one sync command succeeds (`scrape`).
- At least one async command enqueues and reaches terminal state.
- No repeated watchdog reclaim warnings after warm-up.

## Rollback Procedure

Rollback is compose-based and image-based.

1. Stop current stack:

```bash
docker compose --env-file ~/.axon/.env -f docker-compose.yaml down
```

2. Revert deployment inputs:

- previous git revision for compose/docker files
- previous env values if changed
- previous image tags if using tagged images

3. Rebuild/restart:

```bash
docker compose --env-file ~/.axon/.env -f docker-compose.yaml build
docker compose --env-file ~/.axon/.env -f docker-compose.yaml up -d
```

Restart workers and web locally as in the standard deploy procedure.

4. Re-run validation checklist.

## Upgrade Procedure

For code/config upgrades:

1. Review release diff (especially schema, queue names, env vars). Note any new `AXON_HEADLESS_GEMINI_*`, `AXON_LLM_*`, or test infrastructure variables.
2. Re-run `dev-setup.sh` — it backfills new env vars without overwriting existing values.
3. Update `~/.axon/.env` manually for any new required values that need non-default settings.
4. Rebuild and redeploy with standard procedure.
5. Run lifecycle checks:

```bash
./scripts/axon status
./scripts/axon crawl list
./scripts/axon ingest list
```

6. If stale runs from prior version appear, run recover commands.

## Notes for Web UI Paths

- Active UI: `apps/web` (Next.js)
- Core runtime bridge: `axon serve` (`crates/web.rs` + `crates/web/*`) backing `/ws`, `/ws/shell`, `/download/*`, and `/output/*`
- Deprecated piece: only the old standalone static serve page UX

If deploying Next.js:

- Ensure web runtime sees required env vars (`OPENAI_*`, `TEI_URL`, `QDRANT_URL` as needed).
- Validate `/api/pulse/chat` and `/api/omnibox/files` responses.
- MCP servers configured in Web UI are stored in `${AXON_DATA_DIR}/mcp.json` (default: `~/.axon/mcp.json`; fallback: `~/.config/axon/mcp.json`).
- Gemini command selection is strict. Command validation or launch failures surface as errors.
