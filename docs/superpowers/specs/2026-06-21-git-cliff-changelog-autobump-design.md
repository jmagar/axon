# Design: git-cliff changelogs + auto-bump for Axon's release system

- **Date:** 2026-06-21
- **Status:** Approved (design); pending spec review
- **Branch:** `claude/jolly-brahmagupta-ec0a02`
- **Scope:** `xtask` release tooling + `cliff.toml` + `release/components.toml` + per-component `CHANGELOG.md` files + docs. No runtime/library code, no changes to the shipped `axon` binary.

## Problem

Axon already has a mature, custom, per-component release system (`release/components.toml` →
`cargo xtask check-release-versions` → `auto-tag.yml` → per-component release workflows). Two
weak spots remain:

1. **The changelog is worthless.** `ensure_changelog_heading` stamps `"### Changed\n- Release
   version bump."` for every release. The root `CHANGELOG.md` is almost entirely this filler,
   plus a misplaced Keep-a-Changelog preamble. `palette`, `android`, and `chrome` have no
   changelog at all.
2. **Bump level is manual.** A developer must pass `patch|minor|major` to `cargo xtask
   bump-version` and remember to match the `feat!`/`feat`/`fix` convention by hand.

Both are exactly what conventional-commit tooling solves. The repo already uses conventional
commits (`feat:`, `fix:`, `feat(palette):`, `ci:`, `chore:`), and **git-cliff 2.13.1 is already
installed** on the dev host.

## Why not release-plz / release-please

- **release-plz** is Rust/Cargo-only and built around `cargo publish` to crates.io (which Axon
  does not do — it ships GitHub Release binaries). It would cover only the `axon` and
  `axon-palette-tauri` crates (two separate workspaces) and cannot touch `android` or `chrome`,
  leaving two systems. Poor fit.
- **release-please** is the only off-the-shelf tool that fits all four components, but adopting
  it means a workflow-philosophy change (standing release-PR model), a custom updater for
  Android `versionCode`, and porting the parity files to `extra-files`. Large migration, real
  rough edges, for unclear benefit over the working custom system.
- **git-cliff** is the changelog engine release-plz uses internally. Adopting just git-cliff
  (+ a small `xtask` change for auto-bump) fixes both weak spots while keeping the tailored,
  tested multi-component system intact. Lowest risk, highest immediate value.

## Goals

- Real, per-component changelogs generated from conventional commits, for **all four**
  components (`cli`, `palette`, `android`, `chrome`).
- Auto-derive the bump level (and next version) from commits; keep an explicit manual override.
- Zero change to the `auto-tag.yml` → CI gate → tag → release-workflow flow.
- No new CI dependency: `check-release-versions` must not call git-cliff.

## Non-goals

- Replacing the orchestrator (no release-plz/release-please).
- Changing the GitHub Release body (still `generate_release_notes: true` in `release.yml`;
  wiring git-cliff output into the release body is a possible later follow-up, out of scope here).
- Publishing to crates.io.

## Locked decisions

1. **All four components** get a `CHANGELOG.md` generated by git-cliff.
2. **Generated at `bump-version` time** (bump-in-PR), same as today. Changelog section lands in
   the same PR as the version bump; the CI parity gate keeps verifying the heading exists;
   `auto-tag.yml` stays a dumb tagger.
3. **Invoked as a subprocess** to the installed `git-cliff` binary (dev-only tool; no new build
   dependency).
4. **Manual override preserved:** `bump-version <component> [level]` — `level` becomes optional
   and defaults to auto-derived.
5. **git-cliff is the engine for both** the next version (`--bumped-version`) and the changelog
   text, scoped per component, with bump rules in `cliff.toml`.
6. **Regenerate the root CLI `CHANGELOG.md`** cleanly from scoped git history (removes the
   filler + misplaced preamble).
7. **Include the PR-gate suggested-level hint** in v1: `check-release-versions` prints
   git-cliff's suggested bump level for any changed-but-unbumped component.

## Architecture

Four additive parts:

1. **`cliff.toml`** (new, repo root) — single git-cliff config:
   - `[bump]`: `breaking_always_bump_major = true`, `features_always_bump_minor = true` (Axon is
     past 1.0, so breaking → major matches the documented rule).
   - `[git]`: `conventional_commits = true`, `filter_unconventional = true` (drops merge commits
     like `Merge pull request #244 …` and other noise), with concrete `commit_parsers`:
     - `feat` → **Added**
     - `fix` → **Fixed**
     - `perf`, `refactor` → **Changed**
     - `chore`, `ci`, `docs`, `test`, `build`, `style` → **skipped** (not user-facing release
       notes); tunable later in `cliff.toml` if a fuller changelog is wanted.
     - breaking changes (`!` / `BREAKING CHANGE`) are surfaced regardless of type.
   - `[changelog]`: a Keep-a-Changelog body template producing `## [{{version}}] - {{date}}`
     headings with `### Added` / `### Changed` / `### Fixed` (and the other Keep-a-Changelog
     sections as needed) — byte-compatible with the existing root `CHANGELOG.md` format so
     prepends are seamless.
   - Per-component scoping is **not** in `cliff.toml`; it is passed per-invocation by `xtask`
     via `--include-path` / `--tag-pattern` so one config serves all four components.

2. **`release/components.toml`** — add a `changelog_heading` `version_files` entry for
   `palette`, `android`, `chrome` (cli already has one at root `CHANGELOG.md`). No new schema
   fields: scoping is derived from each component's existing `shipping_paths` + `tag_prefix`.
   New changelog paths:
   - `palette` → `apps/palette-tauri/CHANGELOG.md`
   - `android` → `apps/android/CHANGELOG.md`
   - `chrome`  → `apps/chrome-extension/CHANGELOG.md`

3. **`xtask/src/checks/release_versions/cliff.rs`** (+ `cliff_tests.rs`) — the git-cliff glue:
   - `include_paths_for(component)`: each shipping path → `path` (file) and/or `path/**` (dir).
   - `tag_pattern_for(component)`: `^<regex-escaped tag_prefix>` (e.g. cli `^v[0-9]`, palette
     `^palette-v`, android `^android-v`, chrome `^chrome-ext-v`).
   - `next_version(component)`: run `git-cliff --include-path … --tag-pattern … --unreleased
     --bumped-version`, semver-parse / prefix-strip the output.
   - `prepend_changelog(component, next)`: run `git-cliff --include-path … --tag-pattern …
     --unreleased --tag <next> --prepend <changelog>`.
   - `full_history(component)`: same scoping without `--unreleased`, for the initial seed /
     CLI regeneration.

4. **Seeded changelogs** — create `apps/palette-tauri/CHANGELOG.md`,
   `apps/android/CHANGELOG.md`, `apps/chrome-extension/CHANGELOG.md` from each component's
   scoped full history; **regenerate** root `CHANGELOG.md` likewise.

## Per-component scoping (mechanics)

For component X, `xtask` invokes git-cliff with:
- `--include-path <glob>` for each shipping path → only commits touching X's code count.
- `--tag-pattern '^<prefix>'` → X's version baseline is X's tags, not another component's.

Result: "conventional commits touching X since X's last tag." A commit touching both `src/` and
`apps/palette-tauri/` correctly appears in both changelogs (it affected both).

## Auto-bump computation (with guardrails)

`bump`'s signature becomes `level: Option<BumpLevel>`. When `None`:

1. Read current version from `version_source` (existing authoritative reader).
2. `git-cliff … --unreleased --bumped-version` → next version, **semver-parsed/prefix-stripped**.
3. **Validate** it is a strict increment over both the latest tag (existing `latest_tag`) and
   `version_source`, reusing the existing monotonic checks. Error if not.
4. If git-cliff finds no releasable commits → error clearly ("nothing to release for X"), never
   bump blindly.

**Primary implementation risk (verify first):** git-cliff's `--bumped-version` handling for
non-`v` prefixes (`palette-v`, `chrome-ext-v`). **Step 1 of implementation is a spike** that
runs `--bumped-version` against the real tags for each component and confirms the parsed result.
Fallback if git-cliff mishandles a custom prefix: derive only the *level* from git-cliff (diff
its bumped vs latest considered version) and apply that level to `version_source` ourselves.
Either way `xtask` owns the authoritative current version and validates the result.

## Changelog generation + self-trigger fix

- In `bump`, the `ChangelogHeading` `version_files` entry is routed through
  `prepend_changelog(component, next)` instead of the `"Release version bump."` stamp.
- **Self-trigger fix (required for "all four"):** the new component changelogs live *inside*
  shipping paths (e.g. `apps/android/CHANGELOG.md`), so a naive changelog edit would re-trigger
  that component's release. `component_changed_since_ref` must **ignore `CHANGELOG.md`** in
  change detection (basename match within shipping paths) — the same self-trigger guard
  release-please has. The root cli `CHANGELOG.md` is already outside shipping paths, so cli is
  unaffected, but the rule is applied uniformly.

## Seeding / regeneration

- `palette`, `android`, `chrome`: seed `CHANGELOG.md` from `full_history(component)`.
- `cli`: **regenerate** root `CHANGELOG.md` from `full_history(cli)` (locked decision #6).
- After seeding, every component's `CHANGELOG.md` has a heading at its current version, so the
  existing `changelog_heading` parity check passes immediately.

## PR-gate suggested-level hint (locked #7)

In `check-release-versions`, when a component is changed-but-unbumped (the existing failure
case), additionally compute and print git-cliff's suggested level (e.g. `cli: code changed,
suggested bump = minor`). This is advisory output appended to the existing error; it does not
change pass/fail semantics. Because this path runs in CI, it is gated: only invoke git-cliff for
the hint when the binary is present; if git-cliff is unavailable, omit the hint silently (the
underlying changed-but-unbumped error still fires). This preserves "CI does not require
git-cliff."

## git-cliff availability / failure handling

- `bump-version` is a local/PR-time dev step; git-cliff is installed (2.13.1).
- CI's `check-release-versions` does **not** require git-cliff (hint is best-effort, see above).
- If git-cliff is missing during `bump-version`: **fail with an install hint**; never silently
  re-stamp an empty changelog. `ensure_changelog_heading` is retained only as an explicit
  emergency escape hatch behind a `--skip-changelog` flag (off by default).
- Follow-up (outside this repo): add `git-cliff` to the mise config (chezmoi-managed) for fleet
  reproducibility. git-cliff needs full history + tags (`fetch-depth: 0`) — already true where
  bumps run.

## Testing

Sidecar `_tests.rs` convention; files ≤500 lines, functions ≤120.

- **Unit:** `include_paths_for` (dir vs file), `tag_pattern_for` (each prefix, regex-escaping),
  semver extraction/prefix-strip from git-cliff output, `Option<BumpLevel>` parsing, monotonic
  validation, `CHANGELOG.md` change-detection exclusion, `release/components.toml` still parses
  with the new entries.
- **Integration:** reuse the git-fixture harness in `release_versions_tests.rs` — build a temp
  repo with multi-component tags + conventional commits, assert scoped next-version and that the
  prepended section matches the Keep-a-Changelog template, and assert a CHANGELOG-only commit is
  *not* detected as a component change.

## Risks & open items

- **Custom tag-prefix bump parsing** (primary risk) — mitigated by the Step-1 spike + the
  level-derivation fallback above.
- **Template drift** — the body template must match the existing heading format; covered by an
  integration assertion on rendered output.
- **Merge-commit noise** — `filter_unconventional = true` drops `Merge pull request …`; verified
  by the integration fixture including a merge commit.

## Out of scope

- Wiring git-cliff output into the GitHub Release body.
- Any release-plz/release-please adoption.
- mise config change (fleet follow-up, different repo).

## Rollout / sequencing (for the implementation plan)

1. **Spike:** verify `git-cliff --bumped-version` against real per-component tags; lock the
   next-version path (direct vs level-fallback).
2. `cliff.toml` + body template; verify rendered output matches the existing format.
3. `cliff.rs` glue + unit tests.
4. `bump` auto-derive + optional level + changelog routing; `--skip-changelog` escape hatch.
5. `git.rs` `CHANGELOG.md` exclusion + test.
6. `release/components.toml` entries; seed palette/android/chrome changelogs; regenerate cli.
7. PR-gate suggested-level hint (best-effort, CI-safe).
8. `CLAUDE.md` docs update.
9. `just verify` (fmt + clippy + check + test) green.
