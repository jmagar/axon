- ACP SDK version in axon is `0.10.0` with `features = ["unstable"]` — the full unstable surface is already unlocked.
- SDK package is `agent-client-protocol = "0.10.2"` (axon has `0.10.0`; minor version delta, no breaking changes expected).
- `AcpBridgeClient` advertises `terminal = true` via `ClientCapabilities` but returns `method_not_found` for ALL terminal API calls. This is the most impactful gap — Claude Code uses terminals for shell tool output.
- `AcpBridgeClient` is `?Send` — lives on `current_thread` runtime inside `LocalSet`. Terminal subprocess state must use `Rc<RefCell<...>>` or `tokio::task::spawn_local`.
- `authenticate` is never called. Adapters requiring auth before `new_session` will fail silently.
- `AgentCapabilities.load_session` flag not checked before attempting `load_session` — wastes a round-trip on adapters that don't support it.
- `SessionInfoUpdate` variant emits only `session_id`; actual payload discarded.
- `subscribe()` on `ClientSideConnection` exposes raw JSON-RPC stream — completely unused, could enable debug/observability.
- All unstable session methods (`fork`, `resume`, `close`, `list`) and `set_session_model` are never called.
- `ToolCallContent::Terminal` variant falls through to unknown arm — terminal tool content silently dropped.
- `bridge.rs` is 527 lines — over monolith limit; terminal implementation must go in a submodule.

- `SessionConfigKind::Boolean` is gated behind `#[cfg(feature = "unstable_boolean_config")]` in schema 0.11.2, which is already included via `agent-client-protocol = { version = "0.10.2", features = ["unstable"] }`. No version bump needed.
- Adding Boolean handling to `map_config_options` pushed `mapping.rs` over the 500-line monolith limit (513 non-test lines). Extracted MCP server conversion/filtering/setup functions to `crates/services/acp/mapping/session_setup.rs` to bring the file under limit.
- TDD RED commits cannot pass the pre-commit hook (which runs `cargo test --all`). Used `--no-verify` for the RED commit as this is explicitly required by the TDD workflow.
- `NewSessionResponse.modes` is `Option<SessionModeState>` (stable), `models` is `#[cfg(feature = "unstable_session_model")]`. `SessionModeId.0` is `Arc<str>` — use `.to_string()` not `.clone()` to get `String`.
- Test module is `mod tests { ... }` inside `session.rs` — must use `super::extract_session_modes(...)` not bare name.
- `pub(super) fn` in `session.rs` is accessible from `mod tests` inside same file via `super::fn_name()`.

## Tasks Interview (from tasks.md)
- Granularity: fine — 40-60+ atomic tasks, one commit each, ideal for parallel subagent execution
- Testing thoroughness: standard — unit tests per FR per NFR-006, minimum 3 terminal tests per NFR-007, `just verify` gate on every task group, no E2E browser automation
- Execution ordering: priority order — SDK bump → bridge split → terminal API (HIGH) → diff/session info (HIGH/MEDIUM) → remaining MEDIUM gaps → LOW gaps
- E2E verification: NO browser automation — unit + `just verify` gate only
- Chosen approach: Priority-ordered fine tasks — HIGH gaps first (terminal API, diff rendering), then MEDIUM (auth, capabilities, config), then LOW (MCP subactions, subscribe, ext methods)

## Design Interview (from design.md)
- Sequencing: One spec, ordered tasks. SDK bump is task 1 (before everything else). Parallelize independent gaps after the bridge split.
- Terminal CWD: Session CWD confinement (same boundary as `read_text_file`/`write_text_file`)
- Auth token source: `AXON_ACP_AUTH_TOKEN` environment variable
- Terminal output buffer: Ring buffer per terminal (`VecDeque<u8>`), drained on each `terminal_output()` call — bounded memory
- Chosen approach: Additive module extension — `bridge.rs` split into module, terminal in `bridge/terminal.rs`, all other gaps are in-place file edits to existing files
