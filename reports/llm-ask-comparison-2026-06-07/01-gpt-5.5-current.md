Conversation
  You: What is Axon's high-level architecture and which subsystems handle crawl, jobs, vector search, MCP, and web serving?
  Assistant:
Axon’s high-level architecture is a Rust application organized around a CLI entry point, shared core/config utilities, domain subsystems, a typed services layer, and HTTP/MCP/web front ends. Its architecture covers CLI command execution, asynchronous crawl/extract/embed/ingest pipelines, vector storage and retrieval with Qdrant + TEI, and a unified HTTP runtime for `axon serve`, MCP, `/v1/ask`, and direct `/v1` REST routes [S4].

At runtime, `main.rs` and `lib.rs` provide the binary entry and top-level command loop/dispatch, `src/cli/*` contains command handlers and subcommand routing, and `src/core/*` handles config parsing, HTTP safety, content transforms, and logging [S4]. The service layer in `src/services/` is the typed business/service layer consumed by CLI, MCP, and web routes [S8].

The key subsystems are:

| Concern | Subsystem | What it handles |
|---|---|---|
| Crawl | `src/crawl/` / `crawl` | Spider-based crawl engine, render mode strategy, and sitemap backfill [S4][S8] |
| Jobs | `src/jobs/` / `jobs` | SQLite-backed worker runtime and job framework [S4][S8] |
| Vector search / RAG | `src/vector/` / `vector` | Qdrant operations, TEI embedding, hybrid search, semantic query, ask/RAG, retrieve, sources, domains, dedupe, and stats-related Qdrant commands [S6][S8] |
| MCP | `src/mcp/` / `mcp` | MCP server schema, handlers, tool dispatch, stdio entry point, auth, CORS, Streamable HTTP server, and action handlers [S7][S8] |
| Web serving | `src/web/` / `web` | Static setup panel, MCP HTTP, and `/v1/*` REST routes including `/v1/ask`, `/v1/query`, `/v1/scrape`, and job lifecycle routes [S8] |

Axon exposes MCP as a single tool named `axon`, with operation routing through `action` and `subaction` parameters [S2][S3]. The MCP server calls the same services layer used by CLI handlers and web routes [S7]. The unified app service is also described as `axon serve`, exposed on port `8001`, serving the unified Axon HTTP server [S5].

There is also a proposed Cargo workspace refactor that would make these subsystems independent crates: `axon-core`, `axon-crawl`, `axon-vector`, `axon-ingest`, `axon-jobs`, `axon-services`, `axon-mcp`, `axon-web`, and `axon-cli` [S1]. In that proposed dependency graph, `axon-crawl` depends on `axon-core`, `axon-vector` depends on `axon-core`, `axon-jobs` depends on `axon-core`, `axon-crawl`, `axon-vector`, and `axon-ingest`, while `axon-mcp` and `axon-web` depend on `axon-core` and `axon-services` [S1].

### Q2

**Question:** How does Axon's ask pipeline retrieve context and synthesize an answer from indexed documents?

**Answer:**

Conversation
  You: How does Axon's ask pipeline retrieve context and synthesize an answer from indexed documents?
  Assistant:
Axon’s `ask` pipeline starts by embedding the user’s question with TEI, then searches Qdrant for relevant indexed chunks, reranks the results, builds a context window, and sends that context plus the question to the configured LLM for grounded synthesis [S4][S3]. The context-injection docs describe the shared `ask`/`evaluate` flow as five pre-LLM stages: embed, retrieve, filter, rerank, and build context, after which the assembled context is injected into the prompt as `Question: {query}` followed by `Context: {context}` [S1].

For retrieval, Axon encodes the query using `QUERY_INSTRUCTION` from `src/vector/ops/tei/tei_client.rs`, applying the instruction prefix only to query-side embeddings while document chunks are embedded as raw text [S1]. `retrieve_ask_candidates` embeds the full user query and may also embed a distinct keyword query; the full query uses `tei::EmbedInput::query(...)`, while the keyword form uses `tei::EmbedInput::document(...)` to improve recall for exact terms, identifiers, API names, and short domain phrases [S1]. It then queries Qdrant for up to `ask.candidate-limit` candidates, defaulting to `250` [S3][S2].

After retrieval, Axon filters and reranks candidates before selecting context: it applies `ask.min-relevance-score`, default `0.45`, on cosine/dense paths, but skips that threshold in hybrid/RRF named-vector mode because RRF scores are rank-fusion outputs rather than cosine scores [S3][S2]. It reranks by the mode-appropriate score or ordering, takes the top `ask.chunk-limit` chunks, default `20`, then includes additional chunks from the top `AXON_ASK_FULL_DOCS` documents, default `6`, and assembles context up to `AXON_ASK_MAX_CONTEXT_CHARS`, default `300,000` characters [S3][S2]. The context builder can also use knobs such as `ask.backfill-chunks`, `ask.doc-fetch-concurrency`, and `ask.doc-chunk-limit` to control document backfill and context assembly [S2].

For synthesis, Axon calls Gemini headless with the assembled context and question, then applies response-quality gates for citations and policy checks before printing the normalized answer [S3]. The `ask` command is described as RAG-powered Q&A that retrieves relevant chunks from the local Qdrant knowledge base, reranks them by relevance, builds a context window, and calls the configured LLM to generate a grounded answer [S4]. Diagnostics and explain modes can expose retrieval scores, selected chunks, full docs, supplemental context, context character counts, authority ratios, dropped allowlist items, top domains, or per-candidate ranking/context traces; `--explain` skips LLM synthesis [S4].

### Q3

**Question:** What is the services-first contract in Axon, and why should CLI, MCP, and web routes share service functions?

**Answer:**

Conversation
  You: What is the services-first contract in Axon, and why should CLI, MCP, and web routes share service functions?
  Assistant:
The **services-first contract** in Axon is the rule that business logic belongs in `src/services/*`, while CLI, MCP, and web remain thin transport or presentation adapters.[S2] Service functions are the shared API boundary between consumers and infrastructure: they take typed inputs, return typed result structs, avoid stdout side effects, and can be called from any entry point.[S4] Axon’s internal guidance also says CLI handlers should call `src/services::{query,retrieve,ask,summarize,sources,domains,stats,system}` rather than raw native entry points, and MCP handlers plus web routes should call those same service functions and map typed results to their own wire formats.[S3]

CLI, MCP, and web routes should share service functions because Axon previously had duplicated orchestration: CLI embedded business logic in command handlers, MCP reimplemented dispatch with direct function calls, and web shelled out to the CLI binary as a subprocess.[S1] That caused behavior drift between entry points, duplicated orchestration, and a web subprocess overhead of about 200–500ms per command.[S1][S5] The service-layer refactor addresses this by extracting business logic into entry-point-agnostic functions, making CLI, MCP, and web thin shells that format or transport the same underlying results.[S1][S5]

The intended outcome is consistent behavior across surfaces: the services layer is the common boundary for CLI handlers, MCP handlers, and web routes, with lower infrastructure such as vector ops and jobs behind it.[S4] New features are expected to define shared request structs, progress event enums, result structs, and error enums in services, while adapters only map arguments, validate schemas, parse requests, format output, or forward streams/events.[S2] This keeps field names and contracts stable across CLI JSON output, MCP payloads, and web events.[S2]

### Q4

**Question:** How do Axon's SQLite-backed async jobs work, and what does --wait true change for crawl, extract, embed, and ingest commands?

**Answer:**

Conversation
  You: How do Axon's SQLite-backed async jobs work, and what does --wait true change for crawl, extract, embed, and ingest commands?
  Assistant:
Axon’s async jobs are durable, SQLite-persisted jobs processed by in-process workers; there is no message broker, Postgres, or Redis runtime for these jobs [S4]. The four SQLite-backed async job families are Crawl, Extract, Embed, and Ingest, and each has its own SQLite table: `axon_crawl_jobs`, `axon_extract_jobs`, `axon_embed_jobs`, and `axon_ingest_jobs` [S3][S4]. In Docker, the unified `axon` server stores jobs in SQLite and drains them with in-process workers in the same runtime; in local development, `axon serve` runs workers in-process, or individual worker commands such as `axon crawl worker` can be run for focused debugging [S2]. The job lifecycle is `Submitted -> Pending -> Running -> Completed`, with possible `Failed`, `Cancelled`, or `Stale` outcomes [S2].

By default, `--wait false` is fire-and-forget: the command writes or enqueues a SQLite job row and returns the job ID immediately [S2][S5]. For `crawl`, the default async mode writes a SQLite job row and exits without draining other pending crawl rows; workers later run sitemap backfill before any auto-embedding handoff, so sitemap-added pages are visible to the dependent embed job [S5][S6]. Async mode prints one job ID per URL and returns immediately [S5]. The same async-job model applies to Crawl, Extract, Embed, and Ingest, which are the four persisted job families handled by `SqliteJobBackend::new_with_workers` [S4].

`--wait true` changes the command from immediate-return behavior to blocking until completion [S2]. For `crawl`, the command documentation describes `--wait true` as synchronous inline mode: it runs the crawl inline and blocks, and it waits for the submitted crawl plus its explicit dependent embed job if one is created [S5][S6]. The operations guide also shows `crawl --wait true` and `embed --wait true` under “Synchronous (block until done)” examples [S7]. In server mode, `crawl --wait true` polls server job state and does not spawn host-local workers; `--local` is required to force local behavior [S5].

For MCP task workflows, Axon maps task lifecycle methods to the same durable SQLite-backed jobs using task IDs shaped like `axon:<kind>:<job_uuid>` [S1]. Task-mode starts are supported for `crawl.start`, `extract.start`, `embed.start`, and `ingest.start`, while unsupported task/action pairs return `invalid_params` [S1]. Task result retrieval waits for a terminal state, cancellation rejects inactive jobs, list uses a numeric cursor with limit 20 and cap 200, and allowlisted persisted job progress can be forwarded via MCP progress notifications when `_meta.progressToken` is set [S1].

Gaps: The indexed sources clearly define `--wait true` generically and give crawl-specific behavior in detail, but they do not provide equally detailed per-command `--wait true` documentation for extract and ingest. The sources also show embed as a synchronous `--wait true` example, but do not spell out embed’s full blocking implementation details.

### Q5

**Question:** What does Axon's watch scheduler do today, and which watch subcommands are implemented versus only parsed?

**Answer:**

Conversation
  You: What does Axon's watch scheduler do today, and which watch subcommands are implemented versus only parsed?
  Assistant:
Axon’s watch scheduler now auto-fires recurring watches instead of requiring only manual `watch run-now` or HTTP `/v1/watch/{id}/run` execution.[S1] The in-process loop lives in `src/jobs/workers/watch_scheduler.rs`, is spawned by `spawn_workers`, and is active under `axon serve` and `axon mcp`.[S1][S5] On each tick, it leases due enabled watches with `lease_due_watches`, runs them through `run_watch_now_with_pool`, advances `next_run_at` by `every_seconds`, and clears the lease after the run path completes.[S1] It is crash-safe through the stale lease reclamation sweep, and its cadence/lease TTL are controlled by `AXON_WATCH_TICK_SECS` and `AXON_WATCH_LEASE_SECS`, defaulting to 15 seconds and 300 seconds respectively.[S1][S5] `watch create` creates enabled watches, and because pause/resume/delete/update are not implemented, the sources identify no CLI path today to disable, remove, or update a watch after creation.[S5]

Implemented `axon watch` subcommands are `create`, `list`, `run-now`, and `history`.[S2][S3][S5] `axon watch` with no subcommand defaults to `list`.[S2][S3] The parsed-but-stubbed subcommands are `get`, `update`, `pause`, `resume`, `delete`, and `artifacts`; these are present in the CLI schema/help but return “not yet implemented” errors.[S2][S3][S4][S5] `artifacts` takes a `<run_id>` argument, not a watch definition id.[S2][S3][S5]

Gaps: The provided sources disagree on current task-type wording: one command doc says `refresh` is the only dispatched type, while the newer reference doc says `watch` is the only supported type.[S2][S3] The sources also do not show the full current implementation code, only PR notes, docs, help output, and a verification report.

### Q6

**Question:** How does Axon choose between SearXNG and Tavily for search or research, and what happens to discovered results?

**Answer:**

Conversation
  You: How does Axon choose between SearXNG and Tavily for search or research, and what happens to discovered results?
  Assistant:
Axon prefers SearXNG for both `search` and `research` when `AXON_SEARXNG_URL` is set; otherwise it falls back to Tavily using `TAVILY_API_KEY` [S2][S6]. SearXNG must be a self-hosted instance with the `json` output format enabled in `settings.yml` [S1][S6]. Startup/preflight validation accepts either `AXON_SEARXNG_URL` or `TAVILY_API_KEY` for search/research credentials [S1][S7].

For `research`, the selected backend fetches ranked results, then Axon normally fetches full page markdown for the top sources when `AXON_RESEARCH_FULL_CONTENT=true`, falling back to snippets if page fetches fail [S7]. `AXON_RESEARCH_FULL_CONTENT` defaults to `true`, and setting it to `false`/`0`/`no`/`off` makes synthesis use search snippets only, which is faster [S6]. The evidence is wrapped into `evidence_source` blocks with source type/reputation metadata and `instruction_trust=evidence_only`, then a single configured LLM synthesizes the evidence into a summary [S7].

Discovered result URLs are also queued for indexing: `research` queues one bounded crawl job per result URL so sources are indexed asynchronously, and those jobs embed output by default unless `--skip-embed` is used [S7]. The current reference docs say `search` also returns ranked title/URL/snippet results and auto-enqueues one bounded crawl job per result URL for Qdrant indexing, with crawl jobs running asynchronously via the worker pool [S3].

Gaps: Some older command docs still describe Tavily-only behavior and say `search`/`research` do not enqueue crawl jobs [S4][S5][S8], while the newer reference/configuration docs and PR describe SearXNG fallback selection and auto-indexing behavior [S1][S2][S3][S6][S7].

### Q7

**Question:** What are Axon's LLM backend options, and how should AXON_OPENAI_BASE_URL be formatted for openai-compat?

**Answer:**

Conversation
  You: What are Axon's LLM backend options, and how should AXON_OPENAI_BASE_URL be formatted for openai-compat?
  Assistant:
Axon’s `AXON_LLM_BACKEND` supports two synthesis backends: `gemini-headless`, which is the default, and `openai-compat`, for llama.cpp or other OpenAI-compatible `/v1/chat/completions` servers [S2][S4][S5]. The `gemini-headless` backend uses the Gemini CLI, with `AXON_HEADLESS_GEMINI_CMD`, `AXON_HEADLESS_GEMINI_HOME`, and `AXON_HEADLESS_GEMINI_MODEL` as related configuration variables [S1][S2]. The `openai-compat` backend uses `AXON_OPENAI_BASE_URL`, `AXON_OPENAI_MODEL`, and optionally `AXON_OPENAI_API_KEY` [S1][S2][S4].

For `openai-compat`, `AXON_OPENAI_BASE_URL` should be the OpenAI-compatible API root, such as `http://127.0.0.1:8080/v1` [S1][S2]. Do **not** include `/chat/completions`, because Axon appends `/chat/completions` itself [S1][S5]. Include `/v1` when the server exposes chat completions at `/v1/chat/completions` [S5]. `AXON_OPENAI_MODEL` is required when `AXON_LLM_BACKEND=openai-compat` [S1][S2].

The current prefixed variables are `AXON_OPENAI_*`; the legacy unprefixed `OPENAI_*` environment variables were removed in Axon `3.0.0` [S3][S6].

**Question:** What are the important Spider feature-flag gotchas in Axon, especially firewall, balance, and glob?

**Answer:**

Conversation
  You: What are the important Spider feature-flag gotchas in Axon, especially firewall, balance, and glob?
  Assistant:
The important Axon Spider feature-flag gotchas are:

1. **`firewall` must stay disabled for now.** Axon intentionally does **not** enable Spider’s `firewall` feature because `spider_firewall` runs a `build.rs` step that fetches blocklists from `api.github.com` unauthenticated and can panic when CI hits GitHub rate limits; it also does not read `GITHUB_TOKEN`, so CI cannot authenticate that fetch externally [S2][S6][S10]. Axon instead keeps `validate_url()` in `src/core/http/ssrf.rs` as the primary SSRF guard [S2][S6][S10]. The docs say to re-enable only when upstream supports an auth knob [S7][S10]. Note that older inventory snippets still show `firewall` enabled as defense-in-depth [S4][S5], but the newer reference inventory and PR summary say docs were clarified so `firewall` is disabled [S1][S2][S6][S7].

2. **`balance` is not enabled because it silently changes concurrency behavior.** Axon does not enable Spider’s `balance` feature because it silently throttles concurrency with no logging, while Axon manages concurrency itself through performance profiles [S3][S9][S10][S11]. This is especially important operationally because enabling it could make crawl performance look mysteriously slower without an Axon-side explanation [S10][S11].

3. **`glob` is a known breakage trap; do not re-enable it.** Axon does not enable Spider’s native `glob` feature because glob URL patterns change `crawl_establish` to use a budget-aware `is_allowed()` path instead of the default allowed check, and with `with_limit(1)` the first URL can immediately return `BudgetExceeded`, producing zero pages from Chrome crawls [S3][S10][S11]. Axon’s CLI does not rely on Spider-native URL glob patterns, so the feature is excluded [S3][S10]. Axon may have a `--url-glob` config path, but the alignment report says that is processed by Axon before `Website` creation rather than passed through Spider’s native glob handling [S8].

4. **Feature selection is Cargo-level only in Axon.** The local Axon source tree has no `#[cfg(feature = "...")]` gates for these Spider features, so enabling or disabling them happens entirely in the Spider dependency feature list in `Cargo.toml` [S2][S4]. This makes accidental feature additions particularly consequential because behavior changes come from Spider internals rather than explicit Axon call sites [S2].

### Q9

**Question:** How does Axon's hybrid vector search work with Qdrant dense and BM42 sparse vectors, and when does it fall back to dense-only?

**Answer:**

Conversation
  You: How does Axon's hybrid vector search work with Qdrant dense and BM42 sparse vectors, and when does it fall back to dense-only?
  Assistant:
Axon’s hybrid search stores and queries two vector representations in Qdrant: a dense TEI embedding named `"dense"` and a BM42-style sparse vector named `"bm42"`.[S1][S3] New collections are created in Qdrant “Named” mode with `"vectors": {"dense": ...}` plus `"sparse_vectors": {"bm42": {"modifier": "idf"}}`, while legacy collections with unnamed dense vectors remain in “Unnamed” mode.[S1][S7] During indexing, Named-mode upserts include both the dense embedding and the BM42 sparse vector for each point; Unnamed-mode upserts keep the old flat dense-vector format.[S2][S8]

The BM42 sparse side is computed client-side from text: Axon tokenizes text into lowercase alphanumeric terms, filters stopwords, assigns term-frequency weights, and sends sparse `indices` and `values` to Qdrant.[S2][S3] Qdrant applies the IDF correction server-side via the sparse-vector `"modifier": "idf"` setting.[S1][S2] The retrieved sources differ on the exact hashing/bucket implementation: the implementation plan says `hash(term) % 30_000` using `DefaultHasher`, while PR #47 says FNV-1a hashing with `30,522` buckets.[S2][S3]

At query time, `qdrant_hybrid_search` calls Qdrant’s `/points/query` endpoint with two prefetch arms: one querying the `"dense"` vector and one querying the `"bm42"` sparse vector.[S4] Qdrant then fuses the dense and sparse ranked lists using Reciprocal Rank Fusion with `"fusion": "rrf"`, so Axon does not implement separate fusion scoring itself.[S2][S4] The final result count is `limit`, while each prefetch arm fetches `cfg.hybrid_search_candidates` candidates, at least as many as the requested limit.[S4]

Axon falls back to dense-only search when the collection is detected as legacy Unnamed mode, meaning it has the old unnamed dense-vector schema rather than named `"dense"` plus `"bm42"` vectors.[S1][S7] In that case Axon logs that hybrid search is disabled for the collection and continues using dense-only `/points/search` without migrating or modifying the collection.[S5][S7] It also falls back to dense-only when hybrid search is disabled via configuration: setting `AXON_HYBRID_SEARCH=false` reverts search paths to dense-only `/points/search`, while existing Named collections remain usable.[S6]

### Q10

**Question:** What configuration and runtime safeguards prevent Axon ask from overloading smaller-context local models?

**Answer:**

Conversation
  You: What configuration and runtime safeguards prevent Axon ask from overloading smaller-context local models?
  Assistant:
Axon’s main safeguard is **model-tier-aware retrieval defaults**: `ask_max_context_chars`, `ask_chunk_limit`, `ask_candidate_limit`, and `ask_hybrid_candidates` now scale from the configured LLM tier instead of using one large-context profile for every model [S1]. For large models such as Gemini/Claude, the defaults are `1,000,000` context chars, `50` chunks, `250` candidates, and `200` hybrid candidates; for medium models such as Codex they are `400,000`, `28`, `150`, and `120`; for small or unknown models they drop to `40,000`, `10`, `60`, and `60` [S1]. This keeps smaller-context local or unknown models from receiving oversized retrieval windows by default [S1].

Configuration also has bounded knobs for the ask pipeline: `ask.max-context-chars` / `AXON_ASK_MAX_CONTEXT_CHARS` controls total context passed to the LLM and is clamped between `20000` and `1000000`, while `ask.candidate-limit` / `AXON_ASK_CANDIDATE_LIMIT` is clamped between `8` and `300` [S2]. Other context-build controls include `ask.chunk-limit` / `AXON_ASK_CHUNK_LIMIT`, `ask.full-docs` / `AXON_ASK_FULL_DOCS`, `ask.backfill-chunks` / `AXON_ASK_BACKFILL_CHUNKS`, `ask.doc-fetch-concurrency` / `AXON_ASK_DOC_FETCH_CONCURRENCY`, and `ask.doc-chunk-limit` / `AXON_ASK_DOC_CHUNK_LIMIT` [S2]. The docs specifically recommend reducing candidate limits, chunk limits, and context chars to reduce latency, and reducing ask context/candidate limits before increasing worker lanes for high-latency LLM paths [S3].

Axon also protects the context budget at runtime by improving what gets selected: VCS mirror URLs are demoted and cannot win the canonical full-document slot, near-duplicate pages are collapsed, and full documents are inserted all-or-nothing with fallback to chunk coverage if a document exceeds `ask_max_context_chars` [S1]. These safeguards reduce wasted context on mirrors or duplicates, which helps avoid filling a smaller model’s context with redundant or oversized material [S1]. Env/TOML overrides still win, so users can intentionally raise or lower these limits, but the safer model-aware defaults apply when overrides are not set [S1].
