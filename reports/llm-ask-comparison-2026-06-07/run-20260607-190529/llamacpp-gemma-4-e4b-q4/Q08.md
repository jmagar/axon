**MVCC Row Versions and Maintenance Pressure:**
PostgreSQL uses Multiversion Concurrency Control (MVCC) to store rows, which means that updates require adding new versions of rows to tables [S1][S2]. This process generates overhead because it can necessitate new index entries for each updated row, and the subsequent cleanup of old row versions and their associated index entries can be computationally expensive [S1][S2].

**How HOT Reduces Maintenance Pressure:**
The Heap-Only Tuples (HOT) optimization mitigates some of this update overhead when certain conditions are met [S1][S2][S7]. Specifically, HOT can be used if:
1. The update only modifies columns not referenced by the table's indexes (excluding summarizing indexes) [S1][S7].
2. There is enough free space on the page where the old row resides to accommodate the updated version [S1][S7].
When HOT occurs, it provides two main advantages: new index entries are not needed for updated rows (though summary indexes might still require updates), and crucially, old versions of updated rows can be completely removed during normal operations, including `SELECT` queries, rather than waiting for periodic vacuuming [S1][S7].

**Visibility Map and Index-Only Scans:**
The Visibility Map (VM) is a mechanism maintained by `VACUUM` for each heap relation to track which pages contain only tuples known to be visible to all active and future transactions [S4][S10][S11]. This map is used by index-only scans to potentially skip fetching the actual data row from the main heap table [S3][S4][S6]. An index-only scan first checks the VM bit for the corresponding heap page: if the bit is set, the row is known to be visible, allowing the query to return the data using only the index information, thus avoiding a costly heap fetch [S3][S4][S6].
