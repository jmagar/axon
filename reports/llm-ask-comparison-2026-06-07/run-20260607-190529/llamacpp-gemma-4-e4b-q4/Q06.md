  Assistant:
The server sends a `sampling/createMessage` request to the client because the Model Context Protocol (MCP) design requires the client to manage the actual execution and selection of the Large Language Model (LLM) for security and control [S1][S2][S11][S15]. This client-mediated process allows the client to enforce safety and quota restrictions, and it ensures that the server does not need direct access to the LLM provider's API keys [S1][S11]. The request is specifically associated with an originating client request, rather than being initiated standalone, for trust and safety reasons [S15].

The system uses a preference mechanism rather than guaranteeing an exact model because servers and clients might interact with different AI providers that have distinct model offerings [S13][S20]. To accommodate this, MCP implements a system that combines abstract capability priorities with optional model hints [S13][S20].

Model selection is affected by the following components:

*   **Model Hints:** These are suggestions provided as an array of model names or families (e.g., `claude-3-sonnet`) that the server suggests to the client [S2][S13][S20]. Clients can map these hints to equivalent models from different providers [S13][S20].
*   **Capability Priorities:** Servers express needs using three normalized priority values (0 to 1) [S13][S20]:
    *   `costPriority`: Indicates how important minimizing costs is; higher values suggest cheaper models [S13][S20].
    *   `speedPriority`: Indicates the importance of low latency; higher values prefer faster models [S13][S20].
    *   `intelligencePriority`: Indicates the importance of advanced capabilities; higher values prefer more capable models [S13][S20].

Ultimately, the client uses this combination of hints and priorities to make the final model selection from its available options, and this selection is advisory, meaning the client makes the final decision [S13][S20].
