  Assistant:
vLLM builds each KV-cache block hash as `hash(tuple[components])`, where the components are the parent block’s hash value, the tuple of exact token IDs in the current block, and “extra hashes” needed to make the block unique, such as LoRA IDs, multimodal input hashes, and cache salts for multi-tenant isolation [S1][S2].  

The parent hash is what ties the current block to all preceding prefix tokens, so a later block is identified not just by its own tokens but also by the prefix before it [S1][S2]. The current block’s exact tokens are included to reduce potential hash-value collisions [S1][S2]. Extra hashes cover cases where identical placeholder/token sequences are not actually equivalent, such as multimodal prompts where image placeholders must be distinguished by an image hash generated by the frontend image processor [S1][S6]. Cache salts can also be injected into the first block’s hash so that only requests with the same salt can reuse cached KV blocks [S1][S6].  

vLLM only caches full blocks [S1][S2][S5]. In the documented workflow, a block hash is assigned when the block is full, full blocks are added to the cache, and partially filled blocks are not cache hits; for example, with block size 4, a request sharing only 2 of 4 tokens in the third block hits only the first two full cached blocks [S1][S3][S4].  

Before v0.11, the hash key was not guaranteed to be collision-free [S1][S5]. As of v0.11, the default prefix-caching hash algorithm is `sha256`, which the docs say addresses collision risks [S1][S5]. `vllm serve` can still select other algorithms with `--prefix-caching-hash-algo`, including `sha256`, `sha256_cbor`, `xxhash`, and `xxhash_cbor`; the docs warn that non-cryptographically secure algorithms such as xxHash theoretically increase collision risk and can cause undefined behavior or even leak private information in multi-tenant environments [S1][S5].
