**Question:** In MCP sampling, why does the server send a `sampling/createMessage` request through the client instead of directly calling an LLM provider, and how do model hints and priority values affect model selection without guaranteeing an exact model?

  Assistant:
In MCP sampling, the server sends a `sampling/createMessage` request through the client because MCP is designed to let servers request LLM completions while the client keeps control over model access, model selection, and permissions, and so the server does not need its own provider API keys. [S11] This also supports trust-and-safety controls: clients can expose review flows where users inspect or edit prompts, deny requests, and review generated responses before they are returned to the server. [S11][S5] In practice, the server supplies the prompt and optional parameters, while the client decides which model to call, enforces safety or quotas, runs the completion, and returns the result. [S1]

Model hints and priority values are preferences rather than commands because MCP abstracts over different clients and AI providers, and the client may not have access to a requested model or may prefer an equivalent model from another provider. [S2][S7] Priority values are normalized from 0 to 1: higher `costPriority` favors cheaper models, higher `speedPriority` favors lower-latency models, and higher `intelligencePriority` favors more capable models. [S2][S7] Hints let the server suggest specific model names or families, including partial names, and clients may map those hints to equivalent models from different providers. [S4] The client makes the final model selection based on its available models and preferences, and it may ignore the server’s model preferences, so hints and priorities influence selection without guaranteeing an exact model. [S4][S8]
