PostgreSQL’s MVCC stores row updates by adding new row versions to the table, which creates maintenance pressure because each update may also require new index entries and because old row versions and their index entries later have to be removed, which can be expensive. [S1]  

HOT reduces part of that pressure when the update does not change indexed columns, except summarizing indexes, and when there is enough free space on the same heap page to store the updated row version. [S7] In those cases, PostgreSQL can avoid creating new index entries for the updated row, though summary indexes may still need updates. [S7] HOT can also allow intermediate row versions from repeated updates to be removed during normal operation, including `SELECT`s, instead of waiting for periodic vacuum, because indexes continue to refer to the original row version’s page item identifier and that identifier can become a redirect through the HOT chain. [S7]  

The visibility map addresses a different consequence of MVCC: indexes do not store tuple visibility information, so a normal index scan must fetch each matching heap tuple to decide whether it is visible to the current transaction. [S4] Vacuum maintains the visibility map to record pages whose tuples are known to be visible to all active transactions and future transactions until the page is modified. [S4] For an index-only scan, PostgreSQL checks the visibility-map bit for the heap page referenced by a candidate index entry; if the page is marked all-visible, the row can be returned without visiting the heap, but if the bit is not set, PostgreSQL must fetch the heap tuple to check visibility. [S3]  

So, MVCC creates extra table and index cleanup work by accumulating row versions, HOT reduces update-related index churn and some row-version cleanup when its conditions hold, and the visibility map lets index-only scans skip heap fetches only for heap pages known to be all-visible. [S1][S7][S3] This is especially useful because the visibility map is vastly smaller than the heap and can often be cached even when the heap is very large. [S4]
