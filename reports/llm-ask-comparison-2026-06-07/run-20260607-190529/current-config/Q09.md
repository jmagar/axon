dRAID helps resilvering by spreading its internal RAIDZ groups across all child devices, so available disk performance is fully utilized rather than concentrating rebuild work on a single replacement disk or narrow subset of disks [S1]. It also shuffles child vdev mappings so that, regardless of which drive fails, rebuild reads and writes are distributed evenly across all surviving drives [S1]. With distributed hot spare capacity reserved across the children, resilvering to the spare can use the performance of many disks and can substantially reduce the time needed to restore full parity after a failed disk [S10]. The documentation says sequential resilver performance to a distributed hot spare scales with the number of disks divided by the stripe width, `D+P`, which can restore full redundancy in a fraction of the usual time [S3].

The fixed stripe width is what makes fully sequential resilvering possible, because dRAID pads stripes with zeros as needed instead of using RAIDZ’s variable stripe width [S1][S4]. This design has capacity and efficiency costs: with the default `D=8` and 4 KiB sectors, the minimum allocation size is 32 KiB, so smaller logical blocks still consume that allocation granularity [S1][S6]. The source also states that smaller `D` generally increases IOPS, improves compression ratio, and speeds resilvering, but does so at the expense of total usable capacity [S2][S7]. Random IOPS are affected because reads must access all `D` data disks, and delivered random IOPS can be approximated as `floor((N-S)/(D+P)) * single-drive-IOPS` [S2].

Compression is affected because the fixed stripe width can force a relatively large minimum allocation size, and the documentation says that this can reduce the effective compression ratio [S1][S8]. For zvols, the default `volblocksize` is increased when using dRAID to account for that allocation size [S1][S8].

Small-block-heavy dRAID pools may need a mirrored special vdev because the dRAID minimum allocation size can be large for small blocks, and the OpenZFS documentation specifically recommends adding a mirrored special vdev when a dRAID pool will hold a significant amount of small blocks [S1]. A special vdev is dedicated to storing metadata and can optionally store small file blocks and the deduplication table, so it is the documented place to move those small blocks away from the dRAID data layout [S5].
