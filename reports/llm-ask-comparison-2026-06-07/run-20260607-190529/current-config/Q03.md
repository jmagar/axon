  Assistant:
Subnet routers extend a tailnet to specific private subnets so Tailscale devices can reach devices that do not or cannot run the Tailscale client, such as printers, cameras, legacy networks, office LANs, or cloud VPC resources. [S1][S6] Exit nodes solve a different problem: they route outbound internet or non-Tailscale traffic from tailnet devices through a chosen tailnet device, so that traffic appears to come from the exit node’s location. [S1][S8] In short, use a subnet router when you need access *into a private network segment*, and use an exit node when you want a tailnet device’s outbound internet traffic to egress through another device. [S1][S8]

The routing scope is the practical distinction: subnet routers advertise particular CIDR routes and provide access to those private IP ranges, while exit nodes can capture all network traffic that is not already directed to a subnet router or app connector. [S1][S8] Tailscale also describes this as configuring a device to route outbound traffic with an exit node or inbound/private-network access with a subnet router. [S9]

SNAT changes what the destination sees when traffic passes through a subnet router. [S1][S5] By default, Tailscale performs SNAT, also called masquerading, so traffic from a device behind a subnet router is rewritten to appear as if it came from the subnet router itself rather than from the original device. [S1][S5] This reduces routing complexity and enables more transparent communication with the rest of the network, but it hides the original source IP and can prevent traversing multiple networks. [S5][S10] If SNAT is disabled with `--snat-subnet-routes=false` on Linux, the original source IP is preserved, but devices behind the subnet router need a return route for Tailscale traffic, such as routing `100.64.0.0/10` via the subnet router’s LAN IP. [S1][S5]
