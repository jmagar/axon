  Assistant:
To construct a block's hash, vLLM hashes a tuple containing three components: the parent block's hash value, a tuple of the exact tokens within the current block (to minimize collision risks), and any required extra hashes (such as LoRA IDs, multi-modality input hashes, and cache salts) [S1][S2].

vLLM only caches full blocks, assigning the hash metadata once a block is filled so it can immediately be cached and reused by other requests [S1][S3]. While this is the established behavior, the sources do not explicitly explain the technical reasoning or design decisions for why partial blocks are excluded from prefix caching [S1][S2].

Prior to v0.11, the hash keys used by prefix caching were not guaranteed to be collision-free [S1][S5]. As of version v0.11, the default hashing algorithm was changed to `sha256` to address these collision risks [S1][S5]. Users can choose different hashing algorithms via the `--prefix-caching-hash-algo` flag, but choosing non-cryptographic options like `xxhash` theoretically increases the risk of hash collisions, which can lead to undefined behavior or information leaks in multi-tenant environments [S1][S5].
