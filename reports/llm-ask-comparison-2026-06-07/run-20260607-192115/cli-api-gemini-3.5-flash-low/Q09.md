  Assistant:
OpenZFS dRAID's distributed hot spares and fixed stripe width design significantly accelerate resilvering. Distributed hot spares are virtual spare disks created by reserving a fraction of each child drive's capacity [S10]. Because the RAID groups and spares span all children, a precomputed permutation map distributes the rebuild IO (both reads and writes) evenly among all surviving drives [S1][S10]. This allows all disks to participate during a rebuild, increasing available IOPS and bandwidth [S10]. The performance of sequential resilvering to a distributed spare scales with the number of disks divided by the stripe width (D+P) [S3]. Additionally, the fixed stripe width (which pads with zeros as necessary) enables sequential resilvering [S1][S4][S9]. Sequential reconstruction behaves like a traditional RAID rebuild, writing to the replacement device in LBA order, which allows redundancy to be restored quickly before verifying block checksums in a subsequent scrub phase [S9].

The fixed stripe width affects usable capacity, IOPS, and compression due to its strict layout requirements. For example, using the default D=8 data devices and 4 KiB disk sectors establishes a relatively large minimum allocation size of 32 KiB (padding with zeros as necessary) [S1][S6][S8], which reduces overall usable capacity [S9]. This large minimum allocation size can also reduce the effective compression ratio [S1][S6][S8]. In terms of IOPS, read performance is similar to RAIDZ because all D data disks must be accessed for any read [S1][S2]. Delivered random IOPS can be approximated as `floor((N-S)/(D+P)) * <single-drive-IOPS>` [S2]. Generally, a smaller value of D will improve the compression ratio and increase IOPS, but at the expense of total usable capacity [S2][S7].

Because of the large minimum allocation size resulting from the fixed stripe width, pools that process a significant amount of small blocks can experience inefficiencies [S1][S8]. For these small-block-heavy pools, adding a mirrored special vdev is recommended [S1][S2][S8]. The special vdev is dedicated to storing metadata and can optionally store small file blocks directly, bypassing the large minimum allocation overhead of the dRAID vdev [S1][S5][S8].
