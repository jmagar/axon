  Assistant:
MCP sampling routes requests through the client rather than directly calling an LLM provider to keep the client in control of model access, permissions, and security, which eliminates the need for servers to manage or pay for their own LLM API keys and subscriptions [S12][S29][S41]. This design supports a "human-in-the-loop" model, enabling the client to prompt the user to inspect, modify, or approve/deny the request and the resulting generation before it is returned [S12][S14]. It also allows the client to handle tasks like enforcing safety standards and managing quotas [S1].

Because different clients and servers might use entirely different AI providers with distinct offerings, a server cannot dictate a specific model by name; the client may lack access to it or may prefer an equivalent model from a different provider [S13]. As a result, the client has full discretion over the final model selection, and any preferences provided by the server are advisory and can be ignored [S13][S33]. 

To guide the client without forcing an exact model, the server can supply model hints and capability priorities [S13]. Hints are treated flexibly as substrings in order of preference, which the client can map to equivalent models from other providers if the exact suggested model is unavailable [S13]. Normalized priority values from 0 to 1 let the server express its relative needs for `costPriority` (cheaper models), `speedPriority` (lower latency), and `intelligencePriority` (advanced capabilities), allowing the client to evaluate and select the best match from its available options [S13].
