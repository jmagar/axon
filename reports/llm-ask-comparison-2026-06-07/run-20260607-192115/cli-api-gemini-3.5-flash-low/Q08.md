Under PostgreSQL's multiversion concurrency control (MVCC), update queries require new versions of rows to be added to tables [S1][S2]. Because indexes (like B-Tree indexes) treat each row version as an independent object requiring its own index entry, updates can also require new index entries to be created for each updated row [S1][S2][S13]. The resulting "version churn" tuples from update-heavy workloads can accumulate and adversely affect query latency and throughput, and the removal of these old row versions and their corresponding index entries can be expensive [S1][S2][S13].

The Heap-Only Tuples (HOT) optimization reduces this maintenance pressure when an update does not modify any columns referenced by the table's indexes and there is sufficient free space on the page containing the old row [S1][S7]. In these cases, HOT eliminates the need to create new index entries to represent the updated rows [S1][S7]. Furthermore, because indexes continue to reference the page item identifier of the original row version, intermediate row versions that are no longer visible to any transaction can be completely cleaned up and removed during normal operations (including `SELECT`s), rather than waiting for periodic vacuum operations [S1][S7].

The visibility map helps index-only scans avoid heap fetches because PostgreSQL indexes do not store tuple visibility information, normally requiring standard index scans to fetch the heap tuple to check MVCC visibility [S4][S5]. However, the visibility map tracks which heap pages contain only tuples that are known to be visible to all active transactions [S4][S10]. By checking the visibility map first, an index-only scan can skip fetching the heap tuple entirely if the target page's all-visible bit is set [S3][S4]. Since the visibility map is much smaller than the heap and remains cached in memory, this process significantly reduces physical I/O and avoids expensive disk accesses [S3][S4].
