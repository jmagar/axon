## Q07

**Question:** vLLM's automatic prefix caching hashes KV-cache blocks using more than just the current block's tokens. What components go into the block hash, why are only full blocks cached, and what changed around hash collision risk as of v0.11?

  You: vLLM's automatic prefix caching hashes KV-cache blocks using more than just the current block's tokens. What components go into the block hash, why are only full blocks cached, and what changed around hash collision risk as of v0.11?
  Assistant:
vLLM builds each KV-cache block hash as `hash(tuple[components])`, where the components are the parent block’s hash value, the current block’s token tuple, and “extra hashes” needed to make the block unique, such as LoRA IDs, multimodal input hashes, and cache salts for multi-tenant isolation [S1][S2]. The parent hash captures the prefix before the current block, so a later block is identified not just by its own tokens but by the entire preceding chain of blocks [S1][S2]. The exact block tokens are included to reduce potential hash collisions [S1][S2]. For multimodal prompts, vLLM includes an image hash as an extra hash so placeholder tokens for images can be distinguished by the actual image embeddings they represent [S6]. A `cache_salt` can also be injected into the first block’s hash so only requests with the same salt can reuse cached KV blocks [S6][S3].

vLLM only caches full blocks [S1][S2][S5]. The docs show this operationally: a `KVCacheBlock`’s `block_hash` is assigned when the block is full, allocation caches an allocated block immediately if it is already full, and running requests add a block to the cache when the block becomes full [S3][S1]. The example also shows that a partial match inside a block does not produce a cache hit: when the first 10 prompt tokens match but the block size is 4, only the first 8 tokens, i.e. the first two full blocks, hit the cache because the third block matches only 2 of 4 tokens [S1][S4].

Around collision risk, the docs say previous versions did not guarantee the hash key was collision-free, but as of v0.11 the default hashing algorithm is `sha256`, which addresses collision risks [S1][S5]. For `vllm serve`, `--prefix-caching-hash-algo` can select `sha256`, `sha256_cbor`, `xxhash`, or `xxhash_cbor`; the xxHash variants are faster but non-cryptographic, and the docs warn that using a non-cryptographically secure algorithm theoretically increases collision risk, which can cause undefined behavior or leak private information in multi-tenant environments [S1][S5].

Gaps: The sources state that only full blocks are cached and illustrate the behavior, but they do not give a deeper design rationale beyond hash assignment/caching occurring when blocks become full.
