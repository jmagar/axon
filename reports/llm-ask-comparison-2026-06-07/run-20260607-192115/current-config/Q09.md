The fixed stripe width is what enables fully sequential resilvering, because dRAID pads as necessary with zeros rather than using raidz’s variable stripe width.[S1][S4] The tradeoff is that fixed-width allocation affects usable capacity because padding and the chosen data width D impose a minimum allocation granularity; for example, with the default D=8 and 4 KiB sectors, the minimum allocation size is 32 KiB.[S1][S8] It also affects random IOPS because dRAID reads resemble raidz reads in that all D data disks must be accessed for a read, and delivered random IOPS can be approximated as `floor((N-S)/(D+P)) * single-drive-IOPS`.[S2] The `data` setting controls D, and a smaller D generally increases IOPS, improves compression ratio, and speeds resilvering, but reduces total usable capacity.[S2][S7]

The fixed stripe width can reduce compression efficiency because compressed data still has to fit into the relatively large allocation size, and the sources specifically note that the 32 KiB default minimum allocation size can reduce the effective compression ratio.[S1][S8] For zvols on dRAID, OpenZFS increases the default `volblocksize` to account for that allocation size.[S1][S8] If a dRAID pool will contain a significant amount of small blocks, the documentation recommends adding a mirrored special vdev to store those blocks, and a special vdev is defined as a vdev dedicated to metadata and optionally small file blocks and the dedup table.[S1][S5] In practical terms, the mirrored special vdev keeps small-block-heavy workloads from being forced through dRAID’s larger fixed allocation behavior, while mirroring provides the redundancy form indicated by the recommendation.[S1][S5]
