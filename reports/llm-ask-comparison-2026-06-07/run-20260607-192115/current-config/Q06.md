  Assistant:
In MCP sampling, the server sends `sampling/createMessage` through the client because MCP is designed so servers can request LLM completions while the client retains control over model access, model selection, and permissions, and so the server does not need its own provider API keys [S11]. This also supports trust and safety: the client-side application should keep a human in the loop, letting users review or edit prompts before sending and review generated responses before they are delivered back [S11]. In practical terms, the server sends the prompt and optional parameters, while the client decides which model to call, enforces safety or quotas, runs the completion, and returns the response [S1].

Model hints and priority values are preferences, not binding model choices, because servers and clients may use different AI providers with different available models [S2]. A server cannot simply require an exact model name, since the client may not have access to that model or may prefer an equivalent model from another provider [S2]. Hints let the server suggest model names or families, including full or partial names, and clients may map those hints to equivalent models from different providers [S4]. Priority values guide selection by abstract characteristics: higher `costPriority` favors cheaper models, higher `speedPriority` favors lower-latency models, and higher `intelligencePriority` favors more capable models [S7].

The final selection still belongs to the client: it may choose based on the preferences and its available models, and some sources explicitly state the client may ignore the server’s model preferences [S4][S8]. The response can report the actual model used, which may be more specific or different from the hint, such as a request hinting `claude-3-sonnet` and a response reporting `claude-3-sonnet-20240307` [S2].
