OpenZFS dRAID design uses distributed hot spares and fixed stripe width to improve reliability and performance [S1][S4].

### Resilvering Benefits
dRAID utilizes parity declustering, which distributes parity and data groups across all child devices [S1][S4]. Because the stripe width is fixed, the system can perform fully sequential resilvering [S1][S8]. During a rebuild to a distributed hot spare, the IO load is spread evenly across all surviving drives, allowing for faster resilver times and the restoration of full redundancy in a fraction of the time required by traditional methods [S1][S3][S10].

### Impact of Fixed Stripe Width
The use of a fixed stripe width—which involves padding with zeros when necessary—has significant impacts:
*   **Usable Capacity:** The fixed stripe width requires padding, which comes at the expense of some total usable capacity [S1][S9].
*   **IOPS:** The fixed stripe width affects delivered IOPS [S1][S8]. Because all data disks ($D$) must be accessed for any read, random IOPS can be approximated as $\text{floor}((N-S)/(D+P)) \times \text{single-drive-IOPS}$ [S2].
*   **Compression:** Fixed stripe width creates a minimum allocation size (e.g., 32k for a default $D=8$ configuration with 4k sectors) [S1][S6][S8]. This relatively large allocation size can reduce the effective compression ratio [S1][S8].

### Need for a Mirrored Special Vdev
A dRAID pool containing a significant amount of small blocks may benefit from adding a mirrored special vdev [S1][S8]. Because the fixed stripe width results in a larger minimum allocation size, storing many small blocks directly on the dRAID vdev is inefficient; a special vdev allows these smaller blocks to be stored separately, mitigating the impact on capacity and compression efficiency [S1][S8].
