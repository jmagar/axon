PostgreSQL's heap-only tuple (HOT) optimization applies when the following two conditions are met:
* The update does not modify any columns that are referenced by the table's indexes (including expression and partial indexes) [S1][S4].
* There is sufficient free space available on the page that contains the old row version to accommodate the updated row [S1][S4].

When these conditions are satisfied, HOT reduces the following two overheads:
* **Index entry creation:** New index entries are not required to represent the updated rows [S1][S4].
* **Vacuuming requirements:** Instead of requiring periodic vacuum operations, old versions of updated rows can be removed during normal system operations, including `SELECT` queries [S1][S4].
