  Assistant:
MCP sampling utilizes the `sampling/createMessage` request to allow servers to request LLM completions through the client rather than calling an LLM provider directly [S1][S12][S41]. This architecture is designed to keep the client in control of AI model access, security, and permissions, while enabling servers to leverage AI capabilities without needing to manage their own API keys or subscriptions [S12][S19][S41]. Additionally, this design ensures a human-in-the-loop, where the client can present requests and generated responses to the user for review or modification before they are processed or returned to the server [S5][S12][S14].

Model selection is handled through a preference system that balances flexibility with abstraction, as clients and servers may use different providers or have access to different models [S2][S13]. This system uses two components:

*   **Capability Priorities**: Servers express their needs using three normalized values (0–1): `costPriority` (minimizing costs), `speedPriority` (low latency), and `intelligencePriority` (advanced capabilities) [S2][S13].
*   **Model Hints**: Servers provide an array of hints, which are treated as substrings that can match model names flexibly [S13][S20]. Multiple hints may be evaluated in order of preference [S4][S13].

These preferences are advisory rather than mandatory [S13][S20]. Clients make the final model selection based on their available models and specific environment, allowing them to map hints to equivalent models from different providers if the exact model requested is unavailable or if a different model is preferred [S2][S13][S20].
