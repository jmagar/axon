  Assistant:
PostgreSQL utilizes Multiversion Concurrency Control (MVCC) to handle high concurrency, which requires creating a new version of a row whenever an update occurs [S1][S2]. This process creates maintenance pressure because each new row version may necessitate creating new corresponding index entries, and the eventual removal of these old row versions and their index entries—which is handled by periodic vacuuming—can be computationally expensive [S1][S2][S13].

Heap-Only Tuples (HOT) reduce this maintenance pressure by optimizing updates that occur entirely within the heap [S1][S7]. When an update does not modify any columns indexed by the table (excluding summarizing indexes) and there is sufficient free space on the current page, PostgreSQL performs a HOT update [S1][S7]. In these cases, new index entries are not required, and old, dead row versions can be removed during normal operations (such as `SELECT` queries) rather than requiring exclusive reliance on vacuuming [S1][S7].

The visibility map allows some index-only scans to avoid heap fetches because it maintains a record of which heap pages contain only tuples known to be visible to all current and future transactions [S3][S4][S10]. Because PostgreSQL indexes lack tuple visibility information, a standard index scan must fetch the heap tuple to confirm visibility [S4][S6]. However, if the visibility map bit for a page is set, the scan knows all tuples on that page are visible, allowing it to return the data directly from the index [S3][S4][S12]. This prevents unnecessary disk I/O, as the visibility map is much smaller than the heap and is typically cached in memory [S3][S4].
