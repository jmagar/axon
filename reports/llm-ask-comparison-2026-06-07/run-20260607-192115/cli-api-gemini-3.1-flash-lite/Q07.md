  Assistant:
vLLM’s automatic prefix caching implements a hash-based approach where each KV-cache block is uniquely identified by a hash of a tuple of components [S1][S2]. The components included in this hash are:

*   **Parent hash value:** The hash value of the preceding block in the sequence [S2].
*   **Block tokens:** A tuple of the exact tokens contained within the block, included to reduce the likelihood of hash collisions [S2].
*   **Extra hashes:** Additional values required to ensure block uniqueness, such as LoRA IDs, multi-modality input hashes (e.g., image hashes), and cache salts used for multi-tenant isolation [S2][S6].

Regarding why only full blocks are cached, vLLM design notes explicitly state that only full blocks are cached [S2][S5].

As of v0.11, the risk of hash collisions was addressed by making `sha256` the default hashing algorithm [S5]. Previously, the hash key was not guaranteed to be collision-free [S1][S5]. Users can now control the hashing algorithm via the `--prefix-caching-hash-algo` flag, choosing between `sha256` (default), `sha256_cbor`, `xxhash`, and `xxhash_cbor` [S5]. The documentation notes that using algorithms not considered cryptographically secure, such as `xxhash`, theoretically increases the risk of collisions, which could result in undefined behavior or the leakage of private information in multi-tenant environments [S5].
