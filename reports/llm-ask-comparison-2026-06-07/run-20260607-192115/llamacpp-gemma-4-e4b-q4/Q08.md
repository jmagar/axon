PostgreSQL uses Multiversion Concurrency Control (MVCC) to store rows, which requires that updates create new row versions [S2] or [S1]. The inherent overhead of MVCC updates stems from the need to generate new index entries for updated rows and the subsequent, potentially expensive, process of cleaning up old row versions and their index entries [S2], [S1].

Heap-Only Tuples (HOT) is an optimization designed to mitigate this update overhead under specific conditions [S1], [S2]. HOT is possible if an update does not change any columns referenced by the table's indexes and if there is adequate free space on the page containing the old row for the new version [S1], [S2]. When HOT applies, it optimizes updates by eliminating the need to create new index entries for the updated rows, although summary indexes might still require updating [S1]. Furthermore, it allows intermediate, unneeded row versions to be entirely removed during normal operations, including `SELECT`s, rather than necessitating periodic vacuuming [S1], [S7].

The Visibility Map (VM) aids index-only scans by providing information about row visibility, which is distinct from HOT's update mechanics [S3], [S4]. The VM is stored alongside the main table data and tracks which heap pages contain *only* tuples known to be visible to all current and future transactions [S10], [S11]. When an index-only scan executes, it first checks the visibility map bit for the corresponding heap page [S3], [S4]. If the bit is set, the row is known to be visible, allowing the scan to return the data without needing to access the underlying heap page [S3], [S4]. If the bit is not set, the heap entry *must* be visited to confirm visibility, negating the performance benefit of the index-only scan [S3], [S4].
