OpenZFS dRAID uses distributed hot spares and fixed stripe width to enhance data resilience and performance in several ways.

**Resilvering and Distributed Hot Spares:**
dRAID provides integrated distributed hot spares, which allows for faster resilvering compared to traditional RAIDZ implementations [S1], [S2], [S4], [S6]. The design requires shuffling child vdevs using precomputed permutation maps to ensure that the rebuild I/O (both read and write) distributes evenly across all surviving drives, regardless of which drive fails [S1]. When using a distributed spare, the sequential resilver performance scales with the number of disks divided by the stripe width (D+P), which can substantially reduce the time needed to restore full redundancy [S3], [S2], [S4].

**Fixed Stripe Width Effects:**
The fixed stripe width, which necessitates padding with zeros, has significant effects on usable capacity and IOPS [S1], [S4], [S6].
*   **Usable Capacity and Compression:** The fixed stripe width dictates a minimum allocation size; for instance, with default D=8 and 4k sectors, this minimum is 32k [S1], [S6], [S8]. If compression is used, this relatively large allocation size can reduce the effective compression ratio [S1], [S8].
*   **IOPS:** While performance is similar to RAIDZ because all D data disks must be accessed for any read operation, the fixed stripe width significantly affects IOPS [S1], [S4], [S6].
*   **Data Efficiency:** Furthermore, a smaller value for $D$ (data devices per redundancy group) can increase IOPS and improve the compression ratio, though this comes at the cost of total usable capacity [S2], [S7].

**Small-Block-Heavy Pools and Mirrored Special vdev:**
If a dRAID pool is expected to hold a significant amount of small blocks, it is recommended to add a mirrored special vdev to store those blocks [S1], [S8]. This dedicated vdev helps manage the small blocks, which are affected by the large allocation size imposed by the fixed stripe width [S1], [S8].
