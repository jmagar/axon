  Assistant:
The Model Context Protocol (MCP) mandates that the server sends a `sampling/createMessage` request to the client instead of calling an LLM provider directly because this architecture allows the client to maintain control over model access, selection, and permissions, which is crucial for security and privacy, thus eliminating the need for the server to possess any server API keys [S11], [S15]. This client-centric approach also necessitates a human-in-the-loop process where the client presents the request to the user for review before any action is taken [S14], [S15].

Model hints and priority values affect model selection by providing the server with a structured way to express its needs to the client [S13]. The client processes this information to select the model that best fits the requirements from its own available options, which may differ from those the server requested [S13], [S20]. Specifically, the server can suggest preferred models or model families using **model hints**, which are treated as flexible substrings that can match names across different providers [S20]. Additionally, servers specify abstract capability priorities using three normalized values (0-1):

*   **`costPriority`**: Indicates how important minimizing costs is, with higher values preferring cheaper models [S13], [S20].
*   **`speedPriority`**: Indicates how important low latency is, with higher values preferring faster models [S13], [S20].
*   **`intelligencePriority`**: Indicates how important advanced capabilities are, with higher values preferring more capable models [S13], [S20].

The client's final model selection is based on these combined preferences and its local model availability, meaning the hints and priorities are advisory, and the client retains the final authority over model choice [S13], [S20].
