vLLM builds each block hash from three components: the parent block’s hash value, the exact tokens in the current block, and any extra hashes needed to make the block unique, such as LoRA IDs, multimodality input hashes, or cache salts. The docs also note that including the exact block tokens helps reduce potential hash collisions. [S1]

They also state that in previous versions the hash key was not guaranteed to be collision-free; by contrast, non-cryptographic options like `xxhash` are described as increasing collision risk and potentially causing undefined behavior or information leakage in multi-tenant settings. Normalized answer (stored for JSON and follow-up sessions):

vLLM builds each block hash from three components: the parent block’s hash value, the exact tokens in the current block, and any extra hashes needed to make the block unique, such as LoRA IDs, multimodality input hashes, or cache salts. The docs also note that including the exact block tokens helps reduce potential hash collisions. [S1]

The sources say only full blocks are cached, and the `block_hash` is assigned when a block is full and reset when it is evicted. They do not give a separate, explicit justification beyond that design, but the examples and data model consistently treat caching as a full-block operation rather than a partial-block one. [S1][S2]

As of v0.11, the default hashing algorithm changed to `sha256`, which the docs say addresses collision risks. They also state that in previous versions the hash key was not guaranteed to be collision-free; by contrast, non-cryptographic options like `xxhash` are described as increasing collision risk and potentially causing undefined behavior or information leakage in multi-tenant settings. [S1]
