  Assistant:
dRAID helps resilvering by combining integrated distributed hot spares with parity declustering, so rebuild IO is spread evenly across all surviving drives and can be done sequentially to a distributed spare. The docs say this lets dRAID fully utilize disk performance, and that sequential resilvering to a distributed spare can scale with the number of disks divided by the stripe width, which can greatly reduce resilver time and restore full redundancy faster. [S1][S2][S10]

The fixed stripe width is what makes that sequential resilvering possible, but it also reduces usable capacity and IOPS. Because dRAID pads stripes as needed with zeros, the allocation granularity grows; with the default D=8 and 4 KiB sectors, the minimum allocation size is 32 KiB, and the docs note that a smaller D increases IOPS, improves compression ratio, and speeds up resilvering at the expense of total usable capacity. They also say that with compression, this relatively large allocation size can reduce the effective compression ratio. [S1][S6][S8][S7]

For small-block-heavy pools, the recommendation is to add a mirrored special vdev to hold those blocks, because the fixed 32 KiB minimum allocation can be inefficient for lots of small writes. The docs describe a special vdev as a place for metadata and optionally small file blocks, and specifically recommend a mirrored special vdev when a dRAID pool will hold a significant amount of small blocks. [S1][S5]
