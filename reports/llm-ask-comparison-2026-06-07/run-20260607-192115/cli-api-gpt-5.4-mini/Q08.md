In PostgreSQL’s MVCC model, updates create new row versions, can require new index entries for each updated row, and make removal of old versions and their index entries expensive, which creates maintenance pressure. [S1]

HOT reduces part of that pressure when an update does not change any indexed columns and there is enough free space on the same heap page: no new index entries are needed, and older row versions can be removed during normal operation rather than waiting for periodic vacuum, because indexes do not reference page item identifiers. [S1][S7]

For index-only scans, the visibility map records whether a heap page contains only tuples known to be visible to all active transactions; after an index entry is found, the scan checks that map bit, and if the page is all-visible it can skip the heap fetch and return the row using only the index. If the bit is not set, the heap must be visited to verify visibility, so the scan does not avoid the heap access. [S3][S4][S10]
