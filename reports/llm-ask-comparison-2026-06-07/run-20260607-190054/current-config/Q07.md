**Question:** vLLM's automatic prefix caching hashes KV-cache blocks using more than just the current block's tokens. What components go into the block hash, why are only full blocks cached, and what changed around hash collision risk as of v0.11?

  Assistant:
vLLM builds each KV-cache block hash as `hash(tuple[components])`, where the components are the parent block’s hash value, the exact tuple of tokens in the current block, and “extra hashes” needed to make the block unique, such as LoRA IDs, multi-modality input hashes, and cache salts for multi-tenant isolation [S1]. The parent hash is what incorporates the prefix before the current block, so a later block is identified not only by its own tokens but also by the chain of preceding blocks [S1][S2]. For multi-modal inputs, vLLM includes an image hash as an extra hash so that placeholder tokens for an image can be distinguished by the actual image embeddings they correspond to [S6]. A `cache_salt` can also be injected into the first block’s hash so that only requests using the same salt can reuse those cached KV blocks [S6][S3].

vLLM only caches full blocks [S1][S5]. The design shows that a block’s `block_hash` is assigned when the block is full and reset when evicted, and examples show partially full blocks are not cached until they become full [S3][S4]. This also means cache hits occur at full-block granularity: in the example with block size 4, a request sharing 10 initial tokens with a prior request only hits the first 2 blocks, because the third block matches only 2 of 4 tokens [S1].

Around collision risk, earlier versions did not guarantee the hash key was collision-free, but as of v0.11 the default hashing algorithm is `sha256`, which the docs say addresses collision risks [S1][S5]. For `vllm serve`, the hash algorithm can be controlled with `--prefix-caching-hash-algo`, including `sha256`, `sha256_cbor`, `xxhash`, and `xxhash_cbor` [S5]. The docs warn that non-cryptographically secure algorithms such as xxHash can theoretically increase collision risk, which could cause undefined behavior or leak private information in multi-tenant environments [S5].
