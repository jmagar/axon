  Assistant:
dRAID improves resilvering by spreading each internal RAIDZ group across all child disks, using parity declustering so the available disk performance is fully used rather than concentrating rebuild work on a narrow subset of disks. [S1] It also shuffles child vdev mappings with precomputed permutation maps so that, no matter which drive fails, rebuild reads and writes are distributed evenly across surviving drives. [S1] Its distributed hot spares reserve spare capacity across the children, so resilvering to a spare can use the IOPS and bandwidth of all disks and substantially reduce the time needed to restore full parity after a failed disk. [S10] When doing a sequential resilver to a distributed hot spare, performance scales with the number of disks divided by the stripe width, D+P, which can restore full redundancy in a fraction of the usual time. [S3]

The fixed stripe width is what enables fully sequential resilvering, because ordinary RAIDZ’s variable stripe width prevents sequential reconstruction. [S9] The tradeoff is that dRAID must pad as needed with zeros, and this fixed stripe width significantly affects usable capacity and IOPS. [S1] With the default D=8 and 4 KiB disk sectors, the minimum allocation size is 32 KiB, so small logical writes or compressed blocks may still consume a relatively large allocation unit. [S1][S8] That larger allocation size can reduce the effective compression ratio, and OpenZFS increases the default zvol `volblocksize` for dRAID to account for the allocation size. [S1][S8]

IOPS are affected because dRAID read behavior is similar to RAIDZ: for any read, all D data disks must be accessed. [S2] The documentation approximates delivered random IOPS as `floor((N-S)/(D+P)) * <single-drive-IOPS>`, so the chosen data width D and parity width P directly influence random I/O scaling. [S2] The `data` parameter controls the number of data devices per redundancy group, and a smaller D generally increases IOPS, improves compression ratio, and speeds resilvering, but reduces total usable capacity. [S2][S7]

Small-block-heavy dRAID pools may need a mirrored special vdev because dRAID’s relatively large minimum allocation size is inefficient for many small blocks, and the documentation recommends adding a mirrored special vdev to store those blocks when a dRAID pool will hold a significant amount of small blocks. [S1] A special vdev is specifically described as a vdev dedicated to storing metadata and, optionally, small file blocks and the deduplication table. [S5]
