  Assistant:
PostgreSQL’s MVCC creates maintenance pressure because an `UPDATE` adds a new row version to the table rather than overwriting the old one, and that can also require new index entries for each updated row; later, old row versions and their index entries must be removed, which the docs describe as expensive.[S1]

HOT reduces part of that pressure when the update does not modify columns referenced by the table’s indexes, except summarizing indexes, and when there is enough free space on the same page as the old row for the updated row.[S7] In those cases, PostgreSQL can avoid creating new index entries for the updated row, though summary indexes may still need updates.[S7] HOT can also let intermediate row versions be removed during normal operation, including `SELECT`s, rather than waiting for periodic vacuum, because indexes keep referring through the original row-version item identifier and redirects can point to the still-relevant version chain.[S7]

The visibility map addresses a different part of the MVCC cost: visibility checking during reads.[S4] PostgreSQL indexes do not contain tuple visibility information, so a normal index scan must fetch the heap tuple for each matching index entry to check whether the current transaction can see it.[S4] An index-only scan can instead check the visibility-map bit for the relevant heap page; if the bit says the page is all-visible, PostgreSQL knows the row is visible and can skip the heap fetch.[S3][S4] If the bit is not set, the heap tuple still has to be visited to check visibility, so the index-only scan loses that I/O advantage for that entry.[S3]

The reason this helps is that the visibility map is much smaller than the heap and is often cached, so replacing many heap accesses with visibility-map checks can greatly reduce physical I/O when many heap pages are marked all-visible.[S3][S4] Vacuum maintains the visibility map, setting all-visible information for pages whose tuples are known visible to all active and future transactions until the page is modified again.[S4]
