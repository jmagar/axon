### MVCC, Maintenance Pressure, and HOT Reduction

PostgreSQL uses Multiversion Concurrency Control (MVCC) to manage row versions, which is necessary for high concurrency [S2, S1]. A major downside of MVCC for update queries is that updates create new row versions, which in turn can require new index entries and make the removal of old row versions and their index entries expensive [S1, S2].

Heap-Only Tuples (HOT) is an optimization designed to mitigate this update overhead [S1, S2]. HOT is possible if an update only modifies columns *not* referenced by the table's indexes (excluding summarizing indexes) and if there is sufficient free space on the page containing the old row for the new version [S1, S2]. When HOT is applicable, two main benefits arise:
1. New index entries are not required to represent the updated rows, though summary indexes might still need updating [S1].
2. Old versions of updated rows can be completely removed during normal operations (including `SELECT`s), rather than waiting for periodic vacuuming [S1].

### Visibility Map and Index-Only Scans

The visibility map (VM) is a structure associated with each heap relation that tracks which pages contain only tuples known to be visible to all current and future transactions [S10, S11]. The VM is set by `VACUUM` and is cleared by any data-modifying operations on a page [S10, S11].

Index-only scans utilize the VM to potentially avoid fetching the full heap tuple [S4, S6]. A normal index scan must fetch the heap tuple for every matching index entry to verify its visibility under MVCC [S4, S6]. However, when an index-only scan checks the VM and finds that the corresponding page has its "all-visible" bit set, it knows the row is visible and can return the data without the extra heap fetch, leading to a potential I/O saving because the VM is much smaller than the heap it describes [S3, S4].
