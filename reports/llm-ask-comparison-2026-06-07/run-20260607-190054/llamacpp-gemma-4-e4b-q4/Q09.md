The design of OpenZFS dRAID, which includes distributed hot spares and a fixed stripe width, offers specific advantages and consequences for pool operation.

**How Distributed Hot Spares and Fixed Stripe Width Help Resilvering:**
dRAID features integrated distributed hot spares, which allows for faster resilvering while maintaining the benefits of raidz [S1]. Specifically, when performing a sequential resilver to a distributed hot spare, the performance scales with the number of disks divided by the stripe width ($\text{D}+\text{P}$) [S3]. Additionally, the use of a fixed stripe width enables fully sequential resilvering [S4], which helps to reduce the time needed to restore full redundancy after a failure [S3].

**Impact of Fixed Stripe Width on Usable Capacity, IOPS, and Compression:**
The fixed stripe width in dRAID has a significant effect on both usable capacity and IOPS [S1][S4]. For example, using the default $\text{D}=8$ and $4\text{k}$ disk sectors, the minimum allocation size is $32\text{k}$ [S1][S6][S8]. When compression is used, this relatively large allocation size can cause a reduction in the effective compression ratio [S1][S6][S8]. Regarding IOPS, performance is noted to be similar to raidz because any read operation requires accessing all $\text{D}$ data disks [S1][S2]. Furthermore, a smaller value of $\text{D}$ can increase IOPS, improve the compression ratio, and speed up resilvering, though this comes at the cost of total usable capacity [S2][S7].

**Need for Mirrored Special Vdev in Small-Block-Heavy Pools:**
If a dRAID pool is expected to hold a significant amount of small blocks, it is recommended to add a mirrored special vdev to store those blocks [S1].
